Spark读书笔记

2020-09-13

RDD

弹性分布式数据集 (Resilient Distributed Dataset)：是一种分布式的内存抽象，是一种高度受限的共享内存模型，是只读记录分区的集合，只能通过在其他RDD执行确定的转换操作（如 map、join 和 groupBy ）而创建。RDD 没有存储真正的数据，而是含有如何从其他RDD 衍生（即计算）出本 RDD 的相关信息（即Lineage），因此在 RDD 部分分区数据丢失的时候可以从物理存储的数据计算出相应的RDD分区。

RDD 的属性

  • 一组分片（Partition），一个RDD由多个 Partition 组成，每个 Partition 会被一个 task 处理。Partition 是一个逻辑概念，变换前后的新旧分区在物理上可能是同一块内存。创建 RDD 时可指定分片个数，若未指定，则默认值是该程序所分配的 cpu 核数，若是从hdfs 文件创建，则默认为文件的数据块数
  • 一个计算每个分区的函数（Iterator）。RDD的计算是以分片为单位，计算函数都是在对迭代器的复合，不需要保存每次计算的结果。每个RDD都会实现compute函数来计算分区数据
  • RDD 之间的依赖关系。RDD 的每次转换都会生成一个新的 RDD，所以 RDD 之间就会形成类似于流水线一样的前后依赖关系。在部分分区数据丢失时，Spark 可以通过这个依赖关系重新计算丢失的分区数据，而不是对 RDD 的所有分区进行重新计算
  • 一个 Partitioner，即 RDD 的分片函数。决定了父 RDD 和子 RDD 之间的依赖类型。比如 join 操作，若采用协同划分（指定Partitioner 后产生前后一致的分区），两个父RDD与子RDD之间，一个key保证被映射到同一个分区，这样就能形成窄依赖。默认提供两种：HashPartitioner 和 RangePartitioner。Partitioner 值存在于（k,v）类型 RDD 中，否则为none。
  • 一个存储 存取每个 Partition 的优先位置的列表（preferred location）。对于一个 HDFS 文件来说，这个列表保存的就是每个Partition 所在的块的位置。按照“移动数据不如移动计算”的理念，Spark 在进行任务调度的时候，会尽可能地将计算任务分配到其所要处理数据块的存储位置；当 RDD 被缓存，则计算应发送到缓存节点上运行；父 RDD 的优先位置也会决定子 RDD 的位置

RDD 的操作

  1. 创建
    1. 从一个已经存在的Scala集合创建
    2. 由外部存储系统的数据集创建，比如本地文件系统，HDFS 等
  2. 转换：从现有的数据集创建一个新的数据集，例如：map,distinct,union,partitionBy,reduceByKey,cogroup,join
  3. 控制：将 RDD 持久化到内存或磁盘文件系统中，例如：cache, persist
  4. 动作：在数据集上进行计算后，返回一个值给Driver程序，例如：first,count,reduce,sortBy,saveAsTextFile

RDD 中的所有转换都是惰性的，也就是说，它们并不会直接计算结果。它们只是记住这些应用到基础数据集（例如一个文件）上的转换动作。只有当发生一个动作时，这些转换才会真正运行。

RDD之间的关系

RDD 和它依赖的 parent RDD（s）的关系有两种不同的类型：窄依赖（narrow dependency）和宽依赖（wide dependency）。

  1. 窄依赖指的是每一个 parent RDD 的 Partition 最多被子 RDD 的一个 Partition 使用
    由于 RDD 每个 Partition依赖固定数量的 parent RDD（s）的 Partition（s），可以通过一个计算任务来处理窄依赖中一条线上的Partition，Partition 之间（窄依赖中的多条线）相互独立，这些计算任务可以并行执行
    在窄依赖中，节点丢失后的恢复更加高效，因为只有丢失的父分区需要重新计算，并且可以并行在不同节点上计算。
    窄依赖有两种实现：OneToOneDependency、RangeDependency（只被org.apache.spark.rdd.UnionRDD使用）
  2. 宽依赖指的是多个子 RDD 的 Partition 会依赖同一个 parent RDD 的 Partition。
    子 RDD 的 Partition 是 parent RDD 的所有 Partition Shuffle 的结果，因此这两个 RDD 是不能通过一个计算任务来完成的
    宽依赖的实现只有一种：ShuffleDependency
    对应宽依赖的操作，spark 会将中间结果物化到父分区的节点上，以简化数据的故障恢复。

RDD 的存储级别

根据 useDisk(使用磁盘),useMemory(使用内存),useOffHeap(使用其他文件系统),deserialized(反序列化),replication(副本数) 5个参数组合，成为12种存储级别。

spark 默认的存储级别是 不进行序列化直接存储在内存 中，具体的 RDD 数据的存取，请参照 Storage 一节

RDD 的缓存

默认情况下，每一个转换过的 RDD 都会在它执行一个动作时被重新计算。不过也可以使用 persist（或者cache）方法，在内存中持久化一个 RDD，下次计算 RDD 之前，若结果被缓存，则可直接使用结果，避免了重复计算。

RDD的每个 Partition 对应 Storage 模块的一个 Block，只不过 Block 是 Partition 经过处理后的数据。cacheManager 会通过 RDD的ID 和当前计算的 Partition 的ID 向 Storage 模块的 BlockManager 发起查询请求，如果能够获得 Block 的信息，会直接返回 Block 的信息。否则，代表该 RDD 是需要计算的。在计算结束后，计算结果会根据用户定义的存储级别，写入 BlockManager 中。这样，下次就可以不经过计算而直接读取该 RDD 的计算结果。

缓存有可能丢失（存储于内存的数据由于内存不足而被删除、节点宕机），RDD 的缓存的容错机制保证了即使缓存丢失也能保证计算的正确执行。RDD实现了基于 Lineage 的容错机制。RDD 的转换关系，构成了compute chain，可以把这个 compute chain 认为是RDD之间演化的 Lineage。在部分计算结果丢失时，只需要根据这个 Lineage 重算即可。RDD 的各个 Partition 是相对独立的，因此只需要计算丢失的部分即可，并不需要重算全部 Partition。

RDD 的检查点

为了避免缓存丢失重新计算带来的开销，Spark 引入了检查点（checkpoint）机制。缓存是在计算结束后，直接将计算结果通过用户定义的存储级别写入不同的介质。而检查点不同，它是在计算完成后，重新建立一个 Job来计算。

在 Job 结束后，会判断是否需要 checkpoint（在 runjob 方法里，最后有一个doCheckpoint方法）。如果需要，就调用 doCheckpoint 方法首先为数据创建一个目录；然后启动一个新的 Job 来计算该 RDD，并且将计算结果写入新创建的目录；接着创建一个 CheckpointRDD；最后，原始 RDD 的所有依赖被清除，这就意味着 RDD 的转换的计算链（compute chain）等信息都被清除。

由于 checkpoint 操作会对一个RDD 计算两次，所以一般推荐先将 RDD cache 后再进行 checkpoint。

架构

物理架构

Application：用户编写的spark应用程序，包含driver和分布在集群多个节点上运行的executor代码。一个application在执行过程中，会由一个或多个job组成

SparkContext：负责与ClusterManager通信，进行资源申请、任务的分配和监控等

Cluster Manager：集群上获取资源的外部服务，对应于Standalone模式下的Master

worker：集群中任何可以运行 application 代码的节点。主要作用是 向 Master 注册自己，根据 Master 发送的 Application 配置进程环境，启动 ExecutorBackend。

Master：总控进程，Standalone 模式下的主节点，负责管理和分配集群资源（管理所有的Worker，接收 Client 提交的Application，FIFO 调度等待的 Application）。

Executor：执行进程，application运行在worker上的进程，该进程负责运行task

名词解释

Job: RDD 中由行动操作所生成的一个或多个 stage 的组合。

Stage：执行阶段，根据 RDD 之间的依赖关系拆分为多组任务集合。stage 由 DAGScheduler 划分，有 Shuffle Map Stage 和 Result Stage 两种

Task：一个 Stage 是一个任务集（Taskset），task 是分发到 executor 上的工作任务，是执行应用的最小单元

DAGScheduler：面向 Stage 的任务调度器，根据 RDD 的依赖关系划分 Stage 和 taskset，将 taskset 提交给 TaskScheduler

TaskScheduler：面向 task 的调度器，负责 task 的调度执行，高层调度器 DAGScheduler 和任务执行 SchedulerBackend 的桥梁。接收DAGScheduler生成的taskset，把task分发给worker的executor运行。TaskScheduler定义了任务调度相关的实现方法，TaskSchedulerImpl实现了TaskScheduler所有的接口方法。后续的YarnScheduler只需要重写一两个方法即可

SchedulerBackend：负责 application 运行期间和底层资源调度系统的交互，子类根据不同运行模式分为本地 LocalBackend、粗粒度 CoarseGrainedSchedulerBackend、细粒度 MesosSchedulerBackend等

集群启动

RPC

Worker向Master发送的消息主要包含三类：

  • 注册：Worker启动时需要向Master注册，注册时需要汇报自身的信息。
  • 状态汇报：汇报Executor和Driver的运行状态；在Master故障恢复时，需要汇报Worker上当前运行的Executor和Driver的信息。
  • 报活心跳：Worker每隔指定周期会向Master发送报活的心跳。

Master向Worker发送的消息

  • 响应Worker的注册(RegisteredWorker/RegisterWorkerFailed)
  • 控制命令，包括让Worker重新注册、让Worker启动Executor或者Driver、停止Executor和Driver等。

流程

  1. master 启动后，随之启动各 worker。worker 启动时会向所有 master 发送给注册 worker 的消息（RegisterWorker），还要启动一个定时器，在超时的时候进行重新注册。
  2. master 在接到 worker 的注册请求后，如果 master 是 Active 的并且 worker没有注册过，那么 master 会回复 worker消息RegisterWorker，表示 worker注册成功；若注册失败，master 会回复消息 RegisterWorkerFailed，worker接到该消息后打印出错日志并结束启动（由于 worker 会重复多次发送请求，因此退出前需要判断是否注册成功了，如果没有注册成功才会退出；如果已经注册成功了，那么忽略这个消息）。
  3. worker 接收到注册成功后，就可以对外服务了（接收 master 指令等），同时会定时发送心跳信息 heartbeat 给 master，以便master 了解 worker 的实时状态。

应用执行

总览

每个 Spark 的 Application 都是从 SparkContext 开始的，它通过 Cluster Manager 和 Worker上的Executor建立联系，由每个 Executor完成 Application 的部分计算任务。

模块调用

SparkContext 启动后，启动 TaskScheduler 调度器。当 DAGScheduler 将 stage 和 taskset 拆分完毕后，调用TaskScheduler.submitTasks 方法来提交任务，SchedulerBackend 收到 task 后，通过 reviveOffers 方法和 Cluster Manager 交互，取得该 Application 分配到的资源，分配运行资源并启动 worker 上的 executor；最后，由 TaskScheduler 接收任务运行状态，如果task运行完毕，则继续分配，直到 application运行完毕；如果task执行失败，则启动重试机制，并且为执行速度慢的Task启动备份的任务。

不同的 Cluster Master，即资源调度框架的实现模式会有区别，但是 task 的划分和调度都是 DAGScheduler 完成的，资源调度框架在为Application 分配资源后，将 Task 分配到计算的物理单元 Executor 去处理。

RPC 角度

  1. Application 启动，创建 sparkContext，sparkContext 启动过程实例化 SchedulerBackend 对象，该对象启动会创建DriverEndPoint（driver）和 ClientEndPoint（AppClient）。AppClient 启动注册线程向所有 Master 发送 RegisterApplication（注册应用）消息，如果超过20s没有接收到注册成功的消息，那么会重新注册；如果重试超过3次仍未成功，那么本次注册就失败了
  2. Master 收到注册消息后，记录应用信息并将应用加入等待运行列表，后期会根据调度算法（参见应用调度一节）来决定运行哪个应用，发送注册成功（RegisteredApplication）消息给 AppClient；AppClient 收到该消息后，将注册标志置为 true，Master 检测到该标识位变化，完成注册 Application。
  3. Master 找到符合条件的 worker 后，比如，剩余内存、核数符合 task 执行条件，发送 launchExecutor 消息给 worker，通知 worker启动 executor。worker 收到消息后，按照 ExecutorRunner–>CoarseGrainedExecutorBackend 顺序来创建对象，CoarseGrainedExecutorBackend 是executor运行的容器（此时无executor运行）。Worker上的资源信息并不是Worker主动上报到Master的，而是Master主动维护的，因为所有的资源分配都是由Master来完成的，如果Worker启动Executor失败，那么它会发送FAILED的消息给Master，Master收到该消息时再次更新Worker的信息即可
  4. worker发送 ExecutorStateChanged 消息给 Master，通知 Master Executor 容器已创建；Master 收到消息后，向 AppClient 发送 ExecutorAdded 消息
  5. CoarseGrainedExecutorBackend 启动时，发送 RegisterExecutor 消息给 driver
  6. driver 根据 executor 是否已注册，返回 RegisterExecutorFailed/RegisteredExecutor 消息。CoarseGrainedExecutorBackend 收到 RegisteredExecutor 消息后，开始实例化 executor 对象，executor 对象启动完毕后，会定时向 driver 发送心跳，等待接受从driver 发送来的 task
  7. driver 发送 launchTask 给Executor，通知 executor 启动一个 task。executor 会先创建一个 TaskRunner 进程，由该进程执行 task。
  8. TaskRunner 处理 task 完成后，调用 CoarseGrainedExecutorBackend 的 statusUpdate() 方法发送 StatusUpdate 消息给driver；driver 收到消息后，调用 TaskSchedulerImpl 的 statusUpdate 方法，根据任务执行不同结果进行处理，处理完毕后再分配 task 给 executor。

下面主要解析 Job 提交后各组件的处理逻辑。

DAG Scheduler [driver]

创建

SparkContext 在创建时，先创建了 TaskScheduler，后创建了 DAGScheduler，DAGScheduler 保存了TaskScheduler 和 SparkContext 的引用。

DAGScheduler 还包含 MapOutputTrackerMaster（运行在 driver端，管理 Shuffle Map Task 的输出，下游的 Task 可以通过MapOutputTrackerMaster 来获取 Shuffle 输出的位置信息）、BlockManagerMaster（运行在 driver端，管理整个 Job 的 Block 的信息）等对象。

Job 提交

用户提交的 Job（比如 RDD 中的 count方法）最终会调用 DAGScheduler 的 runJob，为这个 Job 生成一个Job ID，并且生成一个JobWaiter 实例来监听 Job 的执行情况：Job 由多个 Task 组成的，只有 Job 的所有 Task 都成功完成，Job 才标记为成功；任意一个 Task 失败都会标记该 Job 失败。

Stage 划分

DAG，Directed Acyclic Graph，有向无环图

根据依赖关系的不同将 DAG 划分为不同的阶段（Stage）：

  • 对于窄依赖，由于 Partition 依赖关系的确定性，Partition 的转换处理就可以在同一个线程里完成，窄依赖被 Spark 划分到同一个Stage
  • 对于宽依赖，由于 Shuffle 的存在，只能在parent RDD（s）Shuffle 处理完成后，才能开始接下来的计算，因此宽依赖就是 Spark 划分 Stage 的依据，即 Spark 根据宽依赖将 DAG 划分为不同的 Stage。

在一个 Stage 内部，每个Partition都会被分配一个计算任务（Task），这些 Task 是可以并行执行的，这些 Task 的执行逻辑完全相同，只是作用于不同的数据。

Stage 之间根据依赖关系变成了一个大粒度的 DAG，DAG 的执行顺序是从前向后的。也就是说，Stage 只有在它没有 parent Stage 或者parent Stage 都已经执行完成后，才可以执行。

Stage 的划分从最后一个 RDD 开始，使用广度优先遍历处理整棵树。从触发 Action 的那个RDD（RDD G）开始，RDD G 依赖于两个RDD，一个是 RDD B，一个是 RDD F。其中先处理 RDD B 还是 RDD F 是随机的。这里假设先处理 RDD B，由于和 RDD B 的依赖是窄依赖，因此 RDD G 和 RDD B 可以划分到一个 Stage（即Stage 3）。再处理 RDD F，由于这个依赖是宽依赖，RDD F 和 RDD G 被划分到不同的 Stage（即 Stage 2 和 Stage 3），其中 RDD F 所在的 Stage 2 是 RDD G 所在的 Stage 3 的 parent Stage。

接下来处理 RDD B 的依赖，由于它依赖的 RDD A 是宽依赖，因此它们属于不同的 Stage（即Stage 1和Stage 3）。这样，RDD B 和 RDD G 同属于Stage 3，而这个 Stage 直接的 parent Stage 有两个，就是 RDD A 和 RDD F 分别属于的两个 Stage（即Stage 1和Stage 2）。接下来的划分采用相同的逻辑，递归进行。

最后，这个 DAG 被划分为三个 Stage，即 RDD A 所在的 Stage 1，RDD C、RDD D、RDD E、RDD F 所在的 Stage 2 和 RDD B 及 RDD G 所在的 Stage 3。其中 Stage 1 和 Stage 2 是相互独立的，可以并发执行；Stage 3 依赖于 Stage 1 和 Stage 2，只有 Stage 1 和 Stage 2 完成计算，它才可以开始计算。

包含最后一个 RDD 的 Stage 就是 Result Stage，其父 Stage 均为 Shuffle Map Stage。

Stage 提交

在 Stage 划分阶段，生成 finalStage 后就会为该 Job 生成一个 ActiveJob，并准备计算这个 finalStage，除了本地运行的 Job，finalStage将被提交，如果它的某些 parent Stage 没有提交，那么递归提交那些未提交（或者叫未计算）的 parent Stage。只有所有的 parent Stage 都计算完成后，才能提交它。

本地运行的条件：

  • spark.localExecution.enabled 设置为true
  • 用户程序显式指定可以本地运行
  • finalStage 没有parent Stage
  • 仅有一个 Partition

当入口的 Stage 运行完成后相继提交后续 Stage，在调度前会先判断该 Stage 所依赖的所有父 Stage 结果是否可用：当一个ShuffleMapStage（父Stage）执行完成，DAGScheduler 会检查该 Stage 的所有 Task 是否都完成了（Executor 执行完 Task 后会发 statusUpdate 消息通知 DAGScheduler 等调度器更新状态），如果存在执行失败的 Task，则重新提交该 Stage；如果父 Stage 结果均可用，则提交该 Stage。

上图的三个 Stage 中，Stage1 和 Stage2 是可以并行的，因此在有计算资源的情况下会首先被提交，并且在这两个 Stage 都计算完成的情况下，提交 Stage 3。

一个 Stage 被提交运行后，DAGScheduler 会根据 Stage 的 Partition 的个数拆分为对应个数 task，这些 task 组成一个 taskset，被提交到 TaskScheduler 上进行处理。对于 Result Stage，它对应的 Task 是 ResultTask，Shuffle Map Stage 对应的 Task 是 ShuffleMapTask。一个 taskset 包含了一个 stage 里所有的 task，这些 task 处理逻辑是一样的，不同的是对应处理的数据。

  • ResultTask：对于最后一个Stage，会根据生成结果的Partition来生成与Partition数量相同的ResultTask。然后，ResultTask会将计算的结果汇报到Driver端

  • ShuffleMapTask：对于非最后的Stage，会根据每个Stage的Partition数量来生成ShuffleMapTask。ShuffleMapTask会根据下游Task的Partition数量和Shuffle的策略来生成一系列文件。

TaskScheduler [driver]

根据不同的部署方式（Standalone、Mesos、YARN或者是Local），会生成不同的 TaskScheduler 和 SchedulerBackend。

以下场景，TaskScheduler 会将 task 分配到 executor 上启动并计算：

  • 有新 task 提交时
  • 有 task 执行失败时
  • Executor 不可用时
  • 某些 task 执行过慢而需要为它重新分配资源时

Task 提交

TaskScheduler 收到 taskset 后，会构建一个 TaskSetManager 对象用来管理 taskset 的生命周期，TaskSetManager 会被放入系统调度池（TaskScheduler 创建的根调度池 rootPool），根据调度算法进行调度，有两种调度策略：

  • FIFO

    rootPool 直接包含了一组 TaskSetManager，根据 Job 优先级（Job 编号越小优先级越高）> Stage 优先级（Stage 编号越小优先级越高）进行调度

  • FAIR

    rootPool 包含了多个下级调度池 Pool，每个 Pool 包含了一组 TaskSetManager，Pool 有如下属性：

    • minShare：最小任务数
    • weight：任务权重
    • SchedulingMode：FAIR或FIFO，Pool中的调度算法

    按照如下顺序来获取哪个 Pool 的哪个 TaskSetManager 会被运行：

    • 某个 TaskSetManager 处于饥饿状态（正在运行的任务数是否小于最小任务），另一个非饥饿，先满足饥饿状态的调度
    • 两个都处于饥饿状态，比较资源比（正在运行的任务数/最小任务数），先满足资源比小的调度
    • 两个都处于非饥饿状态，比较权重比（正在运行的任务数/任务权重），先满足权重比小的调度
    • 以上情况均相同，根据 TaskSetManager 的名称排序

TaskScheduler 会先向 driver 发消息，获取集群中可用的 executor，按照上面算法给 TaskSetManager 按照排序，获取到要调度的 TaskSetManager 后，对 TaskSetManager 中的 task 按照数据本地性分配任务运行节点。

TaskSetManager 的 Locality Levels（数据本地性），按就近原则排序：

  • PROCESS_LOCAL: 数据在同一个 JVM 中，即同一个 executor 上（active 状态的 executors 内存里缓存了 TaskSetManager 的所有 partition 数据）。这是最佳数据 locality。
  • NODE_LOCAL: 数据在同一个节点上。（TaskSetManager 有 task 的优先位置[RDD 的 preferred location]是该节点，且该节点上有 active executor）比如数据在同一个节点的另一个 executor上；或在 HDFS 上，恰好有 block 在同一个节点上。速度比 PROCESS_LOCAL 稍慢，因为数据需要在不同进程之间传递或从文件中读取
  • NO_PREF: 数据从哪里访问都一样快，不需要位置优先（一个 RDD 的某些 partitions 没有优先位置，比如以内存集合作为数据源）
  • RACK_LOCAL: 数据在同一机架的不同节点上。（TaskSetManager 有 task 的优先位置[RDD 的 preferred location]是该机架，且该机架上有 active executor）需要通过网络传输数据及文件 IO，比 NODE_LOCAL 慢
  • ANY: 数据在非同一机架的网络上，速度最慢

通过 computeValidLocalityLevels 方法来计算一个taskSetManager 中的 tasks 都有哪些 locality levels（通过上面括号里的判定方法来计算）。一个 TaskSetManager 的最佳本地性比如是 NODE_LOCAL，则它的可用性集合包括 NODE_LOCAL 及比 NODE_LOCAL 差的级别，即 NODE_LOCAL, NO_PREF, RACK_LOCAL, ANY

对于每一个TaskSetManager，从数据本地性高的 task 开始，遍历所有worker，在符合数据本地性条件的 worker 上尝试给 task分配资源：

  • 分配成功，在 worker 上找到了 executor 可以运行该 task。则更新 taskset 相关记录信息、任务对应 executor 列表等，序列化 task 相关信息并将该 task 放入运行列表中，后续将分发到该 executor 执行
  • 分配失败（该节点没有足够资源运行该任务，比如 worker 上没有足够的 cpu），使用延时调度策略：task 等待一定时间，若在等待时间内，该节点能释放出足够资源，则该 task 会继续在该节点运行，如果还是不足，则 locality level 降低一级，再次进行分配

通过延时调度，让任务运行在有更高级别数据本地性的节点，从而减少磁盘 io 和网络传输。

Executor

Task 执行

CoarseGrainedExecutorBackend 收到 LaunchTask 消息，调用 Executor 的 launchTask 方法进行处理：

  1. 初始化一个 TaskRunner，用于管理 task 运行，将 TaskRunner 放入线程池中去执行
  2. TaskRunner 的 run 方法，向 driver 发送 statusUpdated 消息，通知 driver 任务开始运行，对发送（通过广播变量）过来的 task 本身及依赖 jar 包进行反序列化，调用 Task 的 runtask 方法
  3. 根据 Task 种类分别处理：
    • ShuffleMapTask：若 RDD 已被 cache 或 checkpoint，则直接读取结果。否则进行计算，计算结果保存在本地的 BlockManager 中。最终返回包含数据 location、size 等元数据信息的 MapStatus 对象。
    • ResultTask：进行计算，并返回计算结果
  4. 获得 runtask 方法返回值后，TaskRunner 的 run方法里，将返回值（计算结果）放入 DirectTaskResult 中并序列化，将结果回传到driver 时，会根据结果的大小使用不同的策略：
    • 结果大于 1GB，那么直接丢弃这个结果
    • 结果大小在 [128M-200KB,1GB] 之间，用 taskId 获取一个 blockId，将结果存入 BlockManager 中，返回 driver一个 IndirectTaskResult 对象，该对象包含blockId
    • 结果大小在 [0,128M-200KB) 之间，直接返回序列化后的 DirectTaskResult
  5. TaskRunner.run 方法向 driver 发送 statusUpdated 消息，通知 driver 任务运行结束。driver 收到该消息后，会转给TaskSchedulerImpl 的 statusUpdate 方法根据不同任务状态进行处理：
    • 任务状态是 TaskState.FINISHED，那么调用 enqueueSuccessfulTask 进行处理：若计算结果是 IndirectTaskResult，需要通过blockId 来获取结果，若是 DirectTaskResult，就无需远程获取了
    • 任务状态是 TaskState.FAILED、TaskState.KILLED、TaskState.LOST，调用 enqueueFailedTask 进行处理：
    • TaskState.FAILED，据失败的原因来采取不同的动作，比如如果是因为Task的结果反序列化失败，那么说明task的执行是有问题的，这种task即使重试也不会成功，因此这个 TaskSetManager 会直接失败。如果这些 task 需要重试，且失败次数没有超过阈值（默认4），则会重新提交并调度 task
      • 对于TaskState.LOST，还需要将其所在的 Executor 标记为 FAILED，并且根据更新后的 Executor 重新调度。
  6. 最终任务结束会调用到 DAGScheduler 的 handleTaskCompletion 方法，根据 Task 种类分别处理：
    • ShuffleMapTask：通过 DirectTaskResult/IndirectTaskResult 获取到 MapStatus。
      • 如果该Stage的所有Task都结束了，那么需要将整体结果以 shuffleId 为 key 将 MapStatus 注册到MapOutputTrackerMaster 中。下一个Stage的Task可以提交了，可以通过MapOutputTrackerMaster来获取Shuffle结果的元数据信息，进而从Shuffle数据所在的节点获取数据。
      • 在所有任务都返回的情况下，Stage还有部分数据是空的，则说明部分Task执行失败，需要重新提交这个Stage，重新计算丢失的部分
    • ResultTask：判断 job 是否完成，若完成，则标记改作业已完成，清除 job 依赖的资源并发消息给系统监听总线告知 job 执行完毕

应用调度

当有多个应用程序时，Master 会尽可能将 driver 分散在集群的 worker 节点上启动，采用 FIFO 对 application 进行资源分配，可以通过以下参数来限制 application 使用的 CPU：

  • spark.deploy.defaultCores（每个application能分配到的cpu核数，默认不限制）
  • spark.cores.max（每个application能申请到最大cpu核数）
  • spark.executor.cores（每个executor上启动的cpu核数） 等配置

应用程序的 executor 分布，由 worker 分配策略(spark.deploy.spreadOut 参数)决定:

  • 尽量打散，即将一个 Application 尽可能多地分配到不同的节点。spark.deploy.spreadOut=true，比如，1个 application 有 3个executor，4 个worker，三个 executor 在 3 个 worker 上运行。
  • 尽量集中，即一个 Application 尽量分配到尽可能少的节点。CPU密集型而内存占用比较少的 Application 适合使用这种策略。比如，1个 application 有 3 个 executor，4个worker，三个executor在1个worker上运行。

给某一个应用程序分配资源时，判断worker节点是否能够启动executor：

  • worker上cpu、mem大于executor需要的cpu和mem，executor所需的CPU核数，若没设置则默认为1
  • 该worker上是否允许启动多个executor，若不设置，默认为1

若没有 worker 可以启动 executor，则需要等待其他应用程序释放资源后，再次分配。

Storage

Storage 模块采用的是 Master/Slave 的架构，master（BlockManagerMaster）和 slave（BlockManager）之间使用 rpc 通信。Master负责整个 Application 的 Block 的元数据信息的管理和维护；而 Slave 需要将 Block 的更新等状态上报到 Master，同时接收 Master 的命令，比如删除一个 RDD、Shuffle 相关的数据或者是广播变量；Slave 之间也存在数据传输通道，进行远程的读取和写入。

BlockManager 是 Storage 模块与其他模块交互最主要的类，它提供了读和写 Block 的接口。这里的 Block，实际上就对应了 RDD 中提到的 Partition，每一个 Partition 都会对应一个 Block。每个 Bloc k由唯一的 Block ID 标识，格式是rdd_+rddId+_+partitionId

Master 持有整个 Application 的 Block 的元数据信息，包括Block所在的位置，Block所占的存储空间大小，主要的元数据结构为：

  • blockManagerInfo：保存了BlockManagerId到BlockManagerInfo的映射。BlockManagerInfo保存了executor的内存、数据块使用情况、executor endpoint 的引用
  • blockManagerIdByExecutor：保存了Executor ID到BlockManagerId的映射
  • blockLocations：保存Block是在哪些BlockManager上的Hash Map；由于Block可能在多个executor上都有备份，因此Value是一个HashSet。通过查询blockLocations就可以找到某个Block所在的物理位置

插入、更新数据块：在blockManagerInfo里增加、更新数据块信息；根据blockId在blockLocations里查找该块是否存在，在则更新；若不存在，则进行数据插入

移除数据块：在blockManagerInfo里删除该块信息并更新executor内存使用信息；判断数据块对应的executor是否为空，为空则代表集群中删除了该数据块，在blockLocations中也需要删除该数据块信息

存储交互

Driver中会实例化BlockManagerMaster和BlockManager；executor启动时会实例化BlockManager和BlockTransferService（负责网络数据的传输），BlockManager 负责管理该Executor上的Block，并且向Driver的BlockManager汇报Block的信息和接收来自它的命令

BlockManager会向driver发送RegisterBlockManager消息来把自己的信息注册到BlockManagerMaster中，注册的信息包括blockManagerId（标识了Slave的Executor ID、Hostname和Port），节点的最大可用内存数。BlockManagerMaster接到注册请求后，会将Slave的信息保存到Master端。

应用执行过程中要根据数据块的编号来查询数据块的位置，用getLocations/getLocationsMultipleBlockIds向BlockManagerMaster查询

当数据需要删除时，提交删除消息给slave（BlockManager），slave发起删除，删除driver端元数据信息，还要发消息通知executor删除对应的物理数据

写数据

写入数据完成时，需要将 block 的元数据发送给 BlockManagerMaster，请求更新元数据；若需要进行副本操作，则需要通过netty将数据写入到远程节点上。

写入内存

使用了一个linkedHashMap来保存Block的数据，写入Block数据前会检查当前的内存是否可以容下当前的请求，如果可以，则放入这个Hash Map，期间可能会使用类似FIFO的策略淘汰老的缓存；如果内存超过了当前的最大内存，那么就会返回调用者，如果存储级别还有Disk，那么会将当前的数据通过DiskStore写入Disk，否则缓存将不会被持久化，下次用的话还需要重新计算。

写入磁盘

一般来说，一个逻辑的Block会根据它的BlockId生成的名字映射到一个物理上的文件。这些物理文件会被hash到由spark.local.dir（或者通过SPARK_LOCAL_DIRS）设置的目录中

spark.local.dir 可以设置多个目录。DiskBlockManager 会在spark.local.dir的每个目录下创建一级目录，命名方式是“spark-local-yyyyMMddHHmmss-xxxx”，其中，最后的xxxx是一个随机数，每个一级目录下，会根据需要生成个数至多为spark.diskStore.subDirectories（默认值为64）的子目录，子目录以数字命名（从00到3f，默认）。

DiskBlockManager 根据BlockId文件名的hash值取得其应该存放的一级目录和二级目录，然后将数据写入

读数据

本地读取

  • 内存读取：通过blockId来获取内存中存储的block
  • 磁盘读取：通过文件名（blockId.name）的hash分别确定一级目录和二级目录，找到文件后进行读取，读取后会把这些数据缓存到内存中

远程节点读取

  1. 向BlockManagerMaster发送getLocations查询block的位置列表，driver根据blockId获得该列表
  2. BlockManager根据列表向有block的host上的executor通过netty发送rpc请求获取
  3. 远程节点收到rpc请求后，读取本地数据，使用netty提供的传输通道，将数据传输到请求节点上

HashShuffleManager（基于Hash的Shuffle机制）

为了避免hadoop中多余的排序，提供了基于hash的shuffle写。

Map task中，ShuffleWriter 将原始结果或者聚合后（取决于shuffleDependency是否定义了Aggregator）的结果通过写入n个partition，n 是reduce task的数目，一个partition对应于一个文件， 写入完成后，会将元数据信息写入MapStatus。然后下游的Task可以通过这个MapStatus取得需要处理的数据。

由于每个Shuffle Map Task需要为每个下游的Task创建一个单独的文件，因此文件的数量就是number（shuffle_map_task）*number（reduce_task）。如果Shuffle Map Task是1000，下游的Task是500，那么理论上会产生500000个文件

基于hash的 shuffle 为了解决shuffle 文件数量过多的问题，提出了一个 File Consolidation 的解决方案：

对于运行在同一个Core的Shuffle Map Task，第一个Shuffle Map Task会创建一个文件；之后的就会将数据追加到这个文件上而不是新建一个文件。

因此文件数量就从number（shuffle_map_task）*number（following_task）变成了number（cores）*number（following_task）。当然，如果每个Core都只运行一个Shuffle Map Task，那么就和原来的机制一样了。但是Shuffle Map Task明显多于Core数量或者说每个Core都会运行多个Shuffle Map Task，所以这个实现能够显著减少文件的数量。

SortShuffleManager（基于排序的Shuffle机制）

每个Shuffle Map Task不会为每个Reducer生成一个单独的文件，而是将所有结果写入同一个文件，对应生成一个index文件进行索引：

  • Shuffle Map Task会按照key相对应的Partition ID进行Sort，其中属于同一个Partition的key不会Sort。
  • 如果这个过程内存不够用了，那么这些已经排序的内容会被写入到外部存储，然后在结束的时候将这些不同的文件以及可能存在内存的数据进行归并排序，整合到一个文件里。
  • 为了便于下游的Task获取到其需要的Partition，会生成一个Index文件，去记录不同的Partition的位置信息。
  • 写完数据文件后，将元数据信息写入到MapStatus中

  2. MapOutputTrackerMaster返回数据所在的executor、blockId、数据的大小。MapOutputTracker对数据的位置进行筛选：

      • Hash Based Shuffle的ShuffleBlockManager是org.apache.spark.shuffle.FileShuffleBlockManager：
        consolidateFile：需要根据Map ID和Reduce ID首先获得FileGroup的一个文件，然后根据在文件中的offset和size来获取需要的数据；
        不是consolidateFile：那么根据Shuffle Block ID直接读取整个文件就可以。

容错及HA

Executor 异常

Worker接收到来自Master的LaunchExecutor的消息后，会创建ExecutorRunner。ExecutorRunner会负责启动CoarseGrainedExecutorBackend。CoarseGrainedExecutorBackend会和Driver通信，启动后，会首先向Driver注册Executor（RegisterExecutor）。

  1. 当executor出现异常，比如CoarseGrainedExecutorBackend进程异常退出，ExecutorRunner会捕捉异常并发送ExecutorStateChanged消息给worker
  2. worker收到消息后，更新本地记录的executor状态，同时发送ExecutorStateChanged消息给master
  3. master收到消息后，若发现是executor异常退出，则尝试获取可用的worker节点并启动executor。会尝试10次，超过10次则标记该应用运行失败并在集群中移除该应用

Worker 异常

Worker在运行时会周期性地向Master发送心跳消息。这个周期就默认15s。Master有检测worker超时的线程，当worker进程异常时：

  1. Worker在退出前，会将所有运行在它上面的Executor和Driver Client删除。
  2. Master检测到Worker异常后，根据worker上运行的是Executor和Driver分别进行处理：
    • Executor：将Executor的状态标记为丢失（ExecutorState.LOST），发送ExecutorUpdated消息通知AppClient该executor已lost；把这些executor从对应应用程序的运行列表里删除；处理executor的异常，即上一节的3
    • Driver Client：如果它设置了需要重启，那么需要重新调度来重新启动这个Driver Client，否则直接将它删除，并且将状态设置为DriverState.ERROR。

Master 异常

Master 保存了整个集群的元数据，包含Worker、Application和Driver Client。支持以下几种方式的元数据持久化方式和选举机制：

  • ZooKeeper：实现了基于ZooKeeper的选举机制，元数据信息会持久化到ZooKeeper中。因此在Master故障后，ZooKeeper会在备份的Master中选举出新的Master，新的Master在启动后会从ZooKeeper中获取元数据信息并且恢复这些数据。
  • FILESYSTEM：集群的元数据信息会保存在本地文件系统。而Master启动后则会立即成为Active的Master。如果不考虑机器本身的故障和在设置了Master进程退出后能自动重启的前提下，这种方式也是可以接受的。
  • CUSTOM：用户自定义，需要实现两个接口：持久化元数据和恢复元数据、选举机制，即从几个Standby的Master中选举出一个Master作为集群的管理者
  • NONE：不会持久化集群的元数据，Master在启动后会立即接管集群的管理工作。

可以通过 spark.deploy.recoveryMode进行设 置。如果不设置，那么就默认是 NONE，即没有备份的Master，集群所有历史的元数据信息在Master重启后都会丢失。

在Master的进程启动后，ZooKeeper方式的选举机制会根据自身策略来选举出Leader；对于FILESYSTEM和NONE方式，进程启动后会立即成为Leader。
被选举为Leader的Master，会首先读取集群的元数据信息，如果有读到的数据，那么Master的状态就会变为RecoveryState.RECOVERING，然后开始恢复数据和通知Worker、AppClient和Driver Client，Master已经更改，恢复结束后Master的状态会变成RecoveryState.ALIVE。对于没有读取任何数据的Master，状态会立即变成RecoveryState.ALIVE。
Master只有在状态是RecoveryState.ALIVE时才可以对外服务，包括接受Worker、Application和Driver Client的注册和状态更新等

对于已经有Application运行的集群来说，Master故障恢复的时候，是需要将Application、Worker和Driver Client的元数据恢复的：

  • Application，Master会将它的状态设置为UNKNOWN后通知AppClient
  • Worker，Master会将它的状态设置为UNKNOWN后通知Worker
  • Driver Client，只有在分配给的Worker出现问题时才会重新调度这个Driver Client。因此这里的恢复只是将它加到Master在内存中维护的Driver Client列表中

Master在接到AppClient和Worker的响应后，会将它们的状态从UNKNOWN设置为正常的状态
Master在接到AppClient的响应（消息MasterChangeAcknowledged）和Worker的响应（消息WorkerSchedulerStateResponse）后，会查看是不是所有的Application和Worker的状态都不是UNKNOWN了。如果确认都不是，则代表恢复已经完成。若有部分AppClient或者Worker确实出了问题，长时间没有响应，Master会设置一个超时时间，如果超时之后仍有AppClient或者Worker未响应，那么Master还是会认为恢复已经结束。默认超时时间为60秒，超时后：

  • 将所有未响应的Worker和Application删除
  • 对于未分配Worker的Driver Client（有可能Worker已经死掉），如果 Driver Client 需要重新启动，则重启Driver Client（relaunchDriver）否则，直接删除Driver Client
  • 设置Master的状态为ALIVE

MapReduce的适用范围

任务可以被分解为多个子问题，且这些子问题相对独立，彼此之间不会有牵制，待并行处理完这些子问题后，任务便被解决。

可编程组件

InputFormat

数据切分

按照某个策略将输入数据切分成若干个 split，以便确定 Map Task 个数以及对应的 split。
split，一个逻辑概念，它只包含一些元数据信息，比如数据起始位置、数据长度、数据所在节点等。split 的多少决定了 Map Task 的数目，因为每个 split 会交由一个 Map Task 处理。

client 调用作业 InputFormat 中的 getSplits 函数，将一个文件切分为一个或多个 InputSplit对象， 序列化到文件 job.split 中，InputSplit 的元信息写入到 job.splitmetainfo 文件中。

getSplits 函数：

  • 文件切分算法：解决 一个 maptask 要处理的数据有多大 的问题

    可以看到，当minSize比较小的时候，处理的数据量是小于等于block大小（64M）的，这样可以让一个maptask 不跨 block 处理数据。原因是，如果跨block，两个block可能在不同的节点上，maptask 跨节点处理数据，显然会带来网络开销。

  • host 选择算法

    确定 InputSplit 元数据＜file, start, length, hosts＞，即 InputSplit 所在的文件、起始位置、长度以及所在的host（节点）列表。为了保证任务的本地性，host 一般选择包含该InputSplit数据总量最大的前几个节点。

    此算法也是 maptask 数据本地性的基础，后续会在这些 hosts 上选择并启动 map task 。

为Mapper提供输入数据

给定某个split，将其解析成一个个key/value对。

Mapper

用户定义的对数据处理的方式。

  • 根据用户定义的 JobConf 初始化 Mapper
  • 调用用户编写的 map 函数。MapReduce 框架会通过 InputFormat 从 InputSplit 获取一个个key/value对，交给map() 函数处理：void map（K1 key, V1 value, OutputCollector＜K2，V2＞output, Reporter reporter）throws IOException,OutputCollector 一般用于向磁盘中输出结果

Partitioner

对 Mapper 产生的中间结果进行分片，以便将同一分组的数据交给同一个Reducer处理。会被OutputCollector.collect 方法调用。

public int getPartition（K2 key, V2 value，int numReduceTasks）

两种 Partitioner 实现：

  • HashPartitioner 哈希值分片
  • TotalOrderPartitioner 基于区间的分片，通常用在数据全排序中。按照大小将数据分成若干个区间（分片），并保证后一个区间的所有数据均大于前一个区间数据

Reducer

MapReduce 框架将 从指定的map task 那获取到的数据 处理成＜key, value list＞，调用用户自定义的reduce()函数处理。

OutputFormat

将map() 或者 reduce() 函数产生的 key/value 对写入特定格式的文件中（本地磁盘或HDFS）。
对于推测式任务产生的文件创建、删除、移动，均由该模块完成。

架构组成

RPC协议

JobSubmissionProtocol

  • 提交作业（submitJob）
  • 作业控制：修改其作业优先级（setJobPriority）、杀死一个作业（killJob）、杀死一个任务（killTask）
  • 查看系统状态和作业运行状态：获取集群当前状态，获取某个作业的运行状态，获取所有作业的运行状态等

InterTrackerProtocol

  • 心跳维持：TaskTracker 通过该协议向 JobTracker 汇报所在节点的资源使用情况和任务运行状况，并接收和执行 JobTracker 返回的命令（heartbeat）
  • 从 JobTracker 中获取信息：获取某个作业已经完成的 Task 列表，获取 JobTracker 指定的系统目录（TaskTracker 将作业相关的文件存放到该目录）等

TaskUmbilicalProtocol

  • 周期性汇报及探活：Task每隔3s会调用一次 statusUpdate 函数向 TaskTracker 汇报最新进度。若 Task 在 3s内没有处理任何数据（比如当前记录处理速度太慢），则不再汇报进度，只会调用 ping函数 探测TaskTracker是否活着（在一定时间内，如果某个任务的执行进度和计数器值均未发生变化，则 TaskTracker 认为它处于hang up 状态，会将 Task 杀掉）
  • task 生命周期相关方法：task的初始化，汇报错误及异常，提交处理结果等

JobTracker

JobTracker 是整个集群中唯一的全局“管理者”，涉及的功能包括作业管理、状态监控、任务调度器等。

功能

资源管理模块

资源表示：描述资源的组织方式，采用“槽位”（slot）组织各节点上的资源。slot 相当于任务运行“许可证”。一个任务只有得到该“许可证”后，才能够获得运行的机会，这也意味着，每个节点上的 slot 数目决定了该节点上的最大允许的任务并发度。为了区分 Map Task 和 Reduce Task 所用资源量的差异，slot 又被分为 Map slot 和 Reduce slot 两种，它们分别只能被 Map Task 和 Reduce Task 使用。

资源分配：决定如何将资源分配给各个作业/任务，由一个插拔式的调度器完成。

作业控制模块

作业分解

状态监控（目的：容错、为任务调度提供决策依据）

  • taskTracker 状态
  • job 状态：JobTracker 为每个作业创建一个 JobInProgress 对象以跟踪和监控其运行状态，在作业提交时创建，作业运行完成时销毁。为了区分各个作业，JobTracker会赋予每个作业一个唯一的ID，如 job_201208071706_0009，代表 JobTracker 启动时间、作业提交顺序
  • task 状态：JobTracker将每个 job 拆分成若干个 task，并为每个任务创建一个 TaskInProgress 对象以跟踪和监控其运行状态。每个任务的ID继承了作业的ID，如 task_201208071706_0009_m_000000，添加了 任务类型（map还是reduce）、任务编号
  • TaskAttempt：任务在运行过程中，可能会因为软件Bug、硬件故障等原因运行失败，JobTracker 会按照一定的策略重新运行该任务，每个任务可能会尝试运行多次，直到运行成功或者因超过尝试次数而失败。每运行一次任务即Task Attempt。每个Task Attempt的ID 继承了任务的ID，如 attempt_201208071706_0009_m_000000_0，添加了运行尝试次数

对于某个 task，只要有一个 Task Attempt 运行成功，则相应的 TaskInProgress 对象会标注该 task 运行成功，而当所有的 TaskInProgress 均标注其对应的 task 运行成功后，JobInProgress 对象会标识整个job运行成功。

作业控制：

  • 通过 taskTracker 发来的 heartbeat 来更新 job、task、taskattempt 的状态。JobTracker 使用Map结构保存状态信息：jobs（JobID:JobInProgress）,trackerToTaskMap(trackerID[TaskTracker]:Task ID)
  • 下达命令
    • ReinitTrackerAction：让 TaskTracker 重新初始化。下达该命令原因有如下：
      • jobTracker 对 heartbeat 发送来的消息进行检查，发现了不一致。比如：一个非初次连接的taskTracker，jobTracker中无对应信息
    • LaunchTaskAction ：TaskTracker 接收到该命令后会启动一个子进程运行新任务。
    • KillTaskAction：TaskTracker 收到该命令后会杀掉对应任务、清理工作目录和释放slot。下达该命令原因有如下：
      • 用户调用了“bin/hadoop job-kill-task”或者“bin/hadoop job-fail-task”
      • 在推测执行机制中，当其中一个Task Attempt 执行成功后，另外一个处理相同数据的 Task Attempt将被杀掉
      • 某个 job 运行失败，它的所有 task 将被杀掉
      • TaskTracker 在一定时间内未汇报心跳，则 JobTracker 认为其死掉，它上面的所有Task均被标注为死亡
      • 磁盘空间或者内存不够：MapReduce 框架需采用一定策略杀死若干个 Task 以释放资源。其选择杀死Task的策略是优先选择 Reduce Task，其次是进度最慢的 Task。
    • KillJobAction：TaskTracker接收到该命令后，会杀掉对应作业，清理作业的临时目录。下达该命令原因有如下：
      • 用户调用了 bin/hadoop job-kill 或者 bin/hadoop job-fail
      • job 运行完成，通知 TaskTracker 清理该作业的工作目录
      • job 运行失败，即同一个作业失败的 Task 数目超过一定比例
    • CommitTaskAction：提交任务
后台线程功能
  • expireTrackersThread：主要用于发现和清理死掉的 TaskTracker。每个TaskTracker会周期性地通过心跳向JobTracker汇报信息，而JobTracker会记录每个TaskTracker最近的汇报心跳时间。如果某个TaskTracker在10分钟内未汇报心跳，则JobTracker认为它已死掉，并执行：
    • 从集群中移除：将taskTracker的相关信息从相关数据结构，比如 trackerToTaskMap 中清除
    • 将未运行完成的任务状态标注为 KILLED_UNCLEAN，将它们重新加入任务等待队列中，以便被调度到其他健康节点上重新运行（包含 Reduce Task 的作业中已运行完成的 Map Task 需要重新运行，因为正常的TaskTracker 无法通过HTTP获取死亡 TaskTracker 上的本地磁盘数据）
  • retireJobsThread：主要用于清理长时间驻留在内存中的已经运行完成的作业信息。
  • expireLaunchingTaskThread：用于发现已经被分配给某个TaskTracker但一直未汇报信息（10min）的任务，标记该任务状态为 FAILED。
  • completedJobsStoreThread：将已经运行完成的作业运行信息保存到HDFS上，并提供了一套存取这些信息的API。

稳定性

JobTracker - 场景：jobTracker 挂掉

节点健康和资源分布情况：心跳机制恢复
作业运行时信息恢复：job级别，task级别，记录级别，实现依次变难。

task级别恢复机制

  • 记录关键事件日志（JobHistory）
    • Task：创建、开始运行、运行成功、运行失败、运行被杀死
  • 故障重启后，从日志中恢复作业的运行状态，其中已经运行完成的任务无须再运行，而未开始运行或者运行中的任务需重新运行(RecoveryManager）。

job级别恢复机制：

  • 记录关键事件日志
    • job：提交、创建、开始运行、运行成功、运行失败、运行被杀死
  • 当JobTracker重启后，凡是未运行完成的作业将自动被重新提交到Hadoop中重新运行
TaskTracker - 场景：TaskTracker 挂掉或者任务总是运行失败

  • 超时机制
    如果一种服务在一定时间未响应，则可认为该服务出现了故障，从而启动相应的故障解决方案。使用 expireTrackersThread 线程提供解决方案。

  • 灰名单与黑名单
    这两种名单中的TaskTracker均不可以再接收作业（上面运行的任务可以正常运行结束）。

    • 通过启发式算法推断出存在问题的TaskTracker被加入灰名单，一段时间之后，这些TaskTracker将重新获得一次接收任务的机会。
      • job 在运行过程中记录每个TaskTracker使其失败的Task Attempt数目，一旦该数目超过4（默认值），对应的TaskTracker会被加入该job的黑名单中，而位于黑名单中的TaskTracker将不会再有运行该作业的任何任务的机会。
      • JobTracker将记录每个TaskTracker被作业加入黑名单的次数#blacklist。当某个TaskTracker满足一些规则则，将被加入JobTracker的灰名单中
    • TaskTracker通过心跳将用户设定的脚本的检测结果汇报给JobTracker，一旦发现不健康，JobTracker会将该TaskTracker加入黑名单中，此后不再向其分配任务，直到脚本检测结果为健康

  • Exclude list与Include list

两个名单中的节点均由管理员配置，并可以动态加载生效。

JOB - 场景：部分task运行失败

MapReduce 可允许部分任务失败，而其对应的处理结果不再计入最终的结果。

用户提交作业时可通过配置参数设定允许失败的Map任务和Reduce任务数所占总任务数的百分比。默认情况下，这两个参数值均为0，即只要有一个Map任务或者Reduce任务失败，整个作业便运行失败。

TASK - 场景：运行失败

用户提交作业时可通过这参数设定 Map Task 和 Reduce Task 尝试运行最大次数。默认情况下，这两个值均为4，即每个Task可最多运行4次，如果尝试4次之后仍旧未运行成功，则 TaskInProgress 对象将该任务运行状态标注为FAILED。

fail场景：

  • 人为使Task Attempt失败：用户使用命令“bin/hadoop job-fail-task＜task-id＞
  • 本地文件读写错误：Task运行过程中，由于磁盘坏道等原因，导致文件读写错误。
  • Shuffle阶段错误：Reduce Task从Map Task端远程读取数据过程中出错。
  • 一定时间内未汇报进度：由于程序Bug或者数据格式问题，任务在一定时间间隔（默认10分钟）内未汇报进度。
  • 使用内存量超过期望值：用户提交作业时可指定每个Task预期使用的内存量，如果在运行过程中超过该值，则会运行失败。
  • 任务运行过程中的其他错误：如初始化错误，其他可能的致命错误。

fail和kill的区别

调度策略：一个Task Attempt在某个节点上fail后，调度器便不会再将同一个Task的Task Attempt分配给该节点；而一个Task Attempt kill后，仍可能被调度到同一个节点上运行。
尝试次数：Task容错机制是针对failed Task的，也就是说，任何一个Task允许的失败次数是有限的；而对于killed Task，由于它们是被框架主动杀死的，它们自身并不存在问题，因此会不断尝试运行，直到运行成功。

RECORD - 场景：由于部分数据的原因导致task挂掉

对于一个任务，MapReduce会先尝试运行几次，如果总是失败，则会自动进入skip mode(跳过坏记录)模式。在该模式下，每个Task Attempt不断将接下来要处理的数据区间发送给TaskTracker，再由TaskTracker通过心跳发送给JobTracker，因此，JobTracker时刻保存了尚未处理完成的数据所在区间。如果因某条坏记录导致任务运行失败，JobTracker很容易推断出坏记录所在区间。当重新运行失败任务时，JobTracker将过去识别出的所有坏记录区间“告诉”新的Task Attempt，从而可在运行过程中自动跳过这些坏记录区间。

磁盘 - 场景：磁盘、目录挂掉或者剩余空间不足

TaskTracker：

  • 写入：采用了轮询策略，轮流选择磁盘作为任务的输出目录所在位置。与随机选择和最多剩余空间磁盘优先策略比较，可明显降低产生写热点的可能性。
  • 磁盘空间检查：通过设定参数来检查磁盘空间，决定是否接收新任务和杀死正在运行的任务来释放空间。
  • shuffle数据目录健康状况检查：shuffle数据目录存放在本地磁盘且没有备份，因此一旦损坏或者丢失后，需重新计算。为了降低损坏和丢失的几率，TaskTracker 刚启动时，先检查存放shuffle数据目录的健康状况，将健康目录保存下来以便使用。之后，周期性检查各个目录的健康状况，一旦发现某个正常目录出现故障（比如属性变为只读），则会重新对自己进行初始化。

jobTracker：

通过已经运行完成的任务产生的数据量估算出其他同作业任务需要的磁盘空间，当估算到某个Map/Reduce Task产生的数据量或者输入的数据量超过某个TaskTracker剩余磁盘空间，则不会将该Task分配给它，以防止因为磁盘空间不足而造成任务运行失败。

性能

任务推测机制

问题：在分布式集群环境下，因为程序Bug、负载不均衡或者资源分布不均等原因，会造成同一个作业的多个任务之间运行速度不一致。有些任务的运行速度可能明显慢于其他任务，则这些任务会拖慢作业的整体执行进度。
解决方案：JobTracker 根据一定的规则推测出“拖后腿”的task attempt，并为这样的任务启动一个备份task attempt，让该任务与原始任务同时处理同一份数据，并最终选用最先成功运行完成任务的计算结果作为最终结果。

空间换时间，同时启动多个相同任务处理相同的数据，并让这些任务竞争以缩短数据处理时间。显然，这种方法需要占用更多的计算资源。在集群资源紧缺的情况下，应合理使用该机制，争取在多用少量资源的情况下，减少大作业的计算时间。

调度器

Task类型共分为4种：

  1. setup task（Job-setup Task）：作业初始化标识性任务。该任务运行完后，作业由PREP状态变为RUNNING状态，并开始运行Map Task
  2. map task：map阶段处理数据的任务。其数目及对应的处理数据分片由InputFormat组件确定。
  3. reduce task：Reduce阶段处理数据的任务。
  4. cleanup task（Job-cleanup Task、Task-cleanup Task）：作业结束标志性任务，主要完成一些作业清理工作。一旦该任务运行成功后，作业由RUNNING状态变为SUCCEEDED状态。

四种任务的调度顺序是：

  • 先辅助型任务（不处理实际的数据，job-cleanup task、task-cleanup task、job-setup task）：JobTracker负责调度
  • 再计算型任务（map task，reduce task）：专门的任务调度器对它们进行调度，就是本节所说的调度器。

监听器

  • EagerTaskInitializationListener 对用户提交的作业按照一定顺序进行初始化，优先选择优先级高的作业，当优先级相同时，优先选择提交时间早的作业。
  • JobQueueJobInProgressListener 维护作业的调度顺序，其排序原则跟初始化顺序类似：先按作业优先级排序，然后是作业提交时间，最后是作业ID。

调度机制

使用 assignTasks 实现：

public abstract List＜Task＞assignTasks（TaskTracker taskTracker）

入参是一个TaskTracker对象，出参是为该TaskTracker分配的任务列表。

  • 计算TaskTracker上可用slot数目
  • 分配 task：按照作业调度顺序依次选择一个queue、（选中的queue中的）job、（选中的job中的）task，并将该slot分配给该task

Task 调度策略

  • 数据本地性：根据输入数据与实际分配的计算资源之间的距离将任务分成三类：node-local（输入数据与计算资源同节点），rack-local（同机架）和off-switch（跨机架）。调度器进行任务分配时尽量选择离输入数据近的节点资源。即当一个taskTracker可分配该三类任务，则任务选择优先级从高到低依次为node-local, rack-local和off-switch。
  • 由于计算节点上的 map slot 和 reduce slot 两种资源之间不能混用，因此任务调度器需要分别对 Map Task 和 Reduce Task进行调度。默认情况下，当Map Task完成数目达到总数的5%后，才开始启动Reduce Task（Reduce Task开始被调度）。
  • Map Task 调度：调度器直接调用 JobInProgress 中的 obtainNewMapTask 方法：合法性检查通过后（一个作业在某个节点上失败任务数目超过一定阈值或者该节点剩余磁盘容量不足，则不再将该作业的任何任务分配给该节点），优先选择运行失败的任务，以让其快速获取重新运行的机会（选择失败的任务不再考虑数据本地性），其次是按照数据本地性策略选择尚未运行的任务，最后是查找正在运行的任务，尝试为“拖后腿”任务启动备份任务
  • Reduce Task 调度：调度器直接调用 JobInProgress 中的 obtainNewReduceTask 方法：合法性检查通过后，从未运行task列表中选择任务（不用考虑数据本地性），遍历该列表，选出第一个未曾在该节点上失败过的任务；其次从正在运行的task列表中选择任务，为“拖后腿”任务启动备份任务

TaskTracker

一次心跳

  • 判断是否到达心跳发送时间。
    TaskTracker的心跳间隔由集群规模和任务运行情况共同决定：
    • 集群规模：JobTracker能够根据当前集群规模（TaskTracker数量）动态调整TaskTracker的心跳间隔，并将下一次心跳间隔放到TaskTracker的本次心跳应答中。
    • 任务运行情况：为了提高任务的响应时间和资源利用率，TaskTracker一旦发现存在某个任务运行完成或者失败，就会立即缩短心跳间隔，以便将任务完成或失败的消息告诉JobTracker，进而快速重新分配任务，我们将这种特殊的心跳称为“带外心跳”。
  • 如果TaskTracker刚启动，则检查代码编译版本与JobTracker是否一致。不一致则返回denied，不再发送心跳
  • 检查是否有磁盘损坏。检查存放shuffle数据目录的健康状况。
  • 发送心跳。将当前节点运行时信息，比如资源使用情况、任务运行状态等，通过心跳汇报给JobTracker，同时接收来自JobTracker的各种命令。
  • TaskTracker 接收并执行指令。

执行命令

LaunchTaskAction

本地化是指在TaskTracker上为作业和任务构造一个运行环境，包括创建作业和任务的工作目录，（从HDFS上）下载运行任务相关的文件，如程序jar包、字典文件等，设置环境变量等。在同一个TaskTracker上，由作业的第一个任务完成该作业的本地化工作，后续任务只需进行任务本地化。

作业本地化

通过对相关变量的加锁，保证只有作业的第一个任务为该作业本地化，创建作业相关的目录和文件。

任务本地化

将任务相关的一些配置参数添加到作业配置JobConf中（如果参数名相同，则会覆盖），形成任务自己的配置JobConf并保存到磁盘上，在工作目录中建立指向分布式缓存中所有数据文件的链接，以便能够直接使用这些文件。

启动 Task
  1. 根据任务类型，TaskTracker会调用不同的TaskRunner启动任务。对于Map Task，会调用MapTaskRunner，而Reduce Task则调用ReduceTaskRunner，但任务启动最终均是由TaskRunner.run()方法完成的。
  2. TaskRunner. run()方法首先准备启动任务需要的各种信息，包括启动命令、启动参数、环境变量、标准输出流、标准错误输出流等信息，然后交给JvmManager对象启动一个JVM：
    每个JVM只能同时运行一个任务；
    每个JVM可重复使用以减少启动开销，但某个JVM只限于同一个作业的同类型任务使用。
CommitTaskAction

将一个成功运行完成的Task Attempt结果文件从临时目录“提升”至最终目录。当TaskInProgress中一个任务被提交后，其他任务将被杀死，同时意味着该TaskInProgress运行完成。提交过程采用了两阶段提交协议实现：

第一阶段（准备阶段）：各个参与者（各个TaskTracker）执行完自己的操作后，将状态变为“可以提交”（COMMIT_PENDING），并向协调者发送“准备提交”请求。
第二阶段（提交阶段）：协调者（JobTracker）按照一定原则决定是否允许参与者提交，如果允许，则向参与者发出“确认提交”请求，参与者收到请求后，把“可以提交”状态改为“提交完成”状态（SUCCEEDED），然后返回应答；如果不允许，则向参与者发送“提交失败”请求，参与者收到该请求后，把“可以提交”状态改为“提交失败”状态

  1. Task Attempt处理完最后一条记录后（此时数据写在临时目录下），运行状态由RUNNING变为COMMIT_PENDING，并通过RPC将该状态和任务提交请求发送给TaskTracker；
  2. TaskTracker得知一个Task Attempt状态为COMMIT_PENDING后，立刻缩短心跳间隔，以便快速将任务状态汇报给JobTracker；
  3. JobTracker收到心跳信息后，检查它是否为某个TaskInProgress中第一个状态变为COMMIT_PENDING的Task Attempt，如果是，则批准它进行任务提交，即在心跳应答中添加CommitTaskAction命令，以通知TaskTracker准许该任务提交最终结果；
  4. TaskTracker收到CommitTaskAction命令后，将对应任务加入可提交任务列表commitResponses中；
  5. Task Attempt通过RPC检测到自己位于列表commitResponses中后，进行结果提交，即将数据从临时目录转移到最终目录中，并告诉TaskTracker任务提交完成；
  6. TaskTracker得知任务提交完成后，将该任务运行状态由COMMIT_PENDING变为SUCCEEDED，并在下次心跳中将该任务状态汇报给JobTracker。
KillTaskAction

将一个任务杀死

  1. JobTracker收到来自JobClient的杀死任务请求后，将该任务添加到待杀死任务列表tasksToKill中。
  2. 之后某个时刻，Task Attempt所在的TaskTracker向JobTracker发送心跳，JobTracker收到心跳后，将该任务封装到KillTaskAction命令中，并通过心跳应答发送给TaskTracker。
  3. TaskTracker收到KillTaskAction命令后，将该任务从作业列表runningJobs中清除，并将运行状态从RUNNING转化为KILLED_UNCLEAN，同时通知directoryCleanupThread线程清理其工作目录，释放所占槽位，最后缩短心跳时间间隔，以便将该任务状态迅速通过带外心跳告诉JobTracker。
  4. JobTracker收到状态为KILLED_UNCLEAN的Task Attempt后，将其类型改为task-cleanup task（这种任务的输入数据为空，但ID与被杀Task Attempt的ID相同，其目的是清理被杀Task Attempt已经写入HDFS的临时数据），并添加到待清理任务队列mapCleanupTasks/reduceCleanupTasks中，在接下来的某个TaskTracker的心跳中，JobTracker将其封装到LaunchTaskAction中发送给TaskTracker。
  5. TaskTracker收到LaunchTaskAction后，启动JVM（或重用已启动的JVM）执行该任务。由于该任务属于task-cleanup task，因此它只需清理被杀死的Task Attempt已写入HDFS的临时数据（如果没有则直接跳过），之后其运行状态变为SUCCEEDED，并由TaskTracker通过下一次心跳告诉JobTracker。
  6. JobTracker收到运行状态为SUCCEEDED的Task Attempt后，首先检查它是否位于任务列表tasksToKill中，显然该任务在该列表中，这表明它已经被杀死，于是将其状态转化为KILLED，同时修改相应的各个数据结构。
KillJobAction
  1. 作业的JobInProgress对象将jobKilled变量置为true，并杀死该作业所有的job-setup task、map task和reduce task（注意：job-cleanup task保留）。
  2. 此后假设TaskTracker1第一个向JobTracker汇报心跳，则JobTracker将该作业的job-cleanup task封装成LaunchTaskAction命令，将该TaskTracker对应的已被杀任务封装成KillTaskAction命令，并把这些命令添加到心跳应答中返回给TaskTracker1。
  3. 对于接下来发送心跳的TaskTracker, JobTracker将对应的被杀死任务封装成KillTaskAction命令，并把这些命令添加到心跳应答中返回给这些TaskTracker。
  4. TaskTracker1收到来自JobTracker的命令后，执行相应的操作：若为KillTaskAction命令，则进行的操作与“杀死任务”中的第3步类似，执行完命令后，将作业状态KILLED_UNCLEAN汇报给JobTracker；若为LaunchTaskAction命令，TaskTracker将创建JVM执行该任务，但由于该任务没有需要处理的数据，因此很快可以运行完成，任务完成后，其状态转化为SUCCEEDED，并由TaskTracker通过心跳告诉JobTracker。
  5. 其他TaskTracker收到JobTracker端的KillTaskAction命令后，进行与第4步类似的操作。
  6. JobTracker收到TaskTracker汇报的心跳后，若心跳信息包含任务状态变为KILLED_UNCLEAN，由于任务所属作业已被杀死（jobKilled=true），则将任务由KILLED_UNCLEAN状态转化为KILLED状态；若心跳信息包含job-cleanup task运行成功（状态为SUCCEEDED），则向各个TaskTracker广播KillJobAction命令，以清理作业的工作目录和相关的内存结构（比如runningJobs）。

生命周期

一次任务提交

  1. 通过shell提交hadoop作业（调用 jobclient 进行如下工作）
    1. 将作业文件（jar包，配置文件，普通文件等）上传到hdfs
    2. 调用 InputFormat 的 getSplits 方法生成 job.splitmetainfo 和 job.split 文件
    3. 调用RPC方法 submitJob 将作业提交到 JobTracker 端
  2. JobTracker.submitJob 操作
    1. 为作业创建JobInProgress对象，主要用于跟踪正在运行作业的运行状态和进度
    2. 检查用户是否具有指定队列的作业提交权限
    3. 检查作业配置的内存使用量是否合理
    4. 通知TaskScheduler初始化作业，调度器调用 JobTracker.initJob() 对作业进行初始化（创建TaskInProgress对象）
  3. TaskTracker向JobTracker汇报心跳，其中包含剩余的slot数目和能否接收新任务等信息
  4. 如果该TaskTracker能够接收新任务，则JobTracker调用TaskScheduler对外函数assignTasks为该TaskTracker分配新任务；
  5. TaskScheduler按照一定的调度策略为该TaskTracker选择最合适的任务列表，并将该列表返回给JobTracker；
  6. JobTracker将任务列表以心跳应答的形式返回给对应的TaskTracker；
  7. TaskTracker收到心跳应答后，发现有需要启动的新任务（心跳里有LaunchTaskAction），创建JVM启动任务；

Job状态转换

  • PREP→RUNNING：作业的Setup Task（job-setup Task）成功执行完成。
  • RUNNING→SUCCEEDED:作业的Cleanup Task（job-cleanup Task）执行成功。
  • PREP→FAILED/KILLED:人为使用Shell命令杀死作业，即bin/hadoop job[-kill|-fail]＜jobid＞
  • RUNNING→FAILED:多种情况可导致该状态转移，包括人为使用Shell命令杀死作业，作业的Cleanup/Setup Task运行失败和作业失败的任务数超过了一定比例。
  • RUNNING→KILLED:人为使用Shell命令杀死作业，比如使用“bin/hadoop job-kill＜jobid＞”命令。

Task 状态转换

  • UNASSIGNED→RUNNING：任务初始化状态为UNASSIGNED，当JobTracker将任务分配给某个TaskTracker后，该TaskTracker会为它准备运行环境并启动它，之后该任务进入RUNNING状态
  • RUNNING→COMMIT_PENDING：该状态转换存在于产生最终结果的任务中，当任务处理完最后一条记录后进入COMMIT_PENDING状态，以等待JobTracker批准其提交最后结果。
  • RUNNING→SUCCEEDED:该状态转换只存在于Map Task中，当Map Task处理完最后一条记录后便意味着任务运行成功。
  • COMMIT_PENDING→SUCCEEDED：JobTracker允许提交，TaskAttempt提交完成
  • RUNNING/COMMIT_PENDING→KILLED_UNCLEAN：TaskTracker收到来自JobTracker的KillTaskAction命令。
  • RUNNING/COMMIT_PENDING→FAILED_UNCLEAN：多种情况下会导致该状态转移，参见task fail场景
  • UNASSIGNED→FAILED/KILLED:人为杀死任务。
  • KILLED_UNCLEAN/FAILED_UNCLEAN→FAILED/KILLED:一旦任务进入KILLED_UNCLEAN/FAILED_UNCLEAN状态，接下来必然进入FAILED/KILLED状态，以清理已经写入HDFS上的部分结果。
  • SUCCEEDED→KILLED:一个TIP已有一个Task Attempt运行完成，而备份任务也汇报成功，则备份任务将被杀掉或者用户人为杀死某个Task，而TaskTracker刚好汇报对应Task执行成功。
  • SUCCEEDED/COMMIT_PENDING→FAILED:Reduce Task从Map Task端远程读取数据时，发现数据损坏或者丢失，则将对应Map Task状态标注为FAILED以便重新得到调度。

Map Task

  • read阶段：Map Task 调用InputFormat组件的getSplits函数，从InputSplit中解析出一个个key/value。

  • Map阶段：将解析出的key/value交给用户编写的map()函数处理，并产生一系列新的key/value。

  • Collect阶段：map() 函数数据处理完成后，会调用OutputCollector.collect()函数输出结果。

    • 如果Reduce Task数目为0，则直接将结果写入HDFS中作为最终结果
    • Reduce Task数目不为0，会调用Partitioner.getPartition()函数获取记录的分区号partition，将生成的key/value分片，然后将三元组＜key, value, partition＞传递给MapOutputBuffer.collect() 函数做进一步处理。

  • Spill阶段：即“溢写”。

    MapOutputBuffer内部使用了一个环形缓冲区暂时存储用户输出数据，当缓冲区使用率达到一定阈值后，由线程 SpillThread 将数据写到一个临时文件中：

    1. 利用快速排序算法对缓冲区数据进行排序，排序方式是，先按照分区编号partition进行排序，然后按照key进行排序。这样，经过排序后，数据以分区为单位聚集在一起，且同一分区内所有数据按照key有序
    2. 按照分区编号由小到大依次将每个分区中的数据写入任务工作目录下的临时文件output/spillN.out（N表示当前溢写次数）中。如果用户设置了Combiner（ 把形如这样<a,1>,<a,2>key值相同的数据进行计算，计算规则与reduce一致，比如：当前计算是求key对应的值求和，则combine操作后得到<a,3>这样的结果 ），则写入文件之前，对每个分区中的数据进行一次聚集操作。
    3. 将分区数据的元信息写到内存索引数据结构SpillRecord中，其中每个分区的元信息包括在临时文件中的偏移量、压缩前数据大小和压缩后数据大小。如果当前内存中索引大小超过1 MB，则将内存索引写到文件output/spillN.out.index中。

    环形缓冲区使得Map Task的Collect阶段和Spill阶段可并行进行。

  • merge 阶段：当所有数据处理完成后，Map Task对所有临时文件以分区为单位进行合并，对于某个分区，它将采用多轮递归合并的方式：每轮合并io.sort.factor（默认为100）个文件，并将产生的文件重新加入待合并列表中，对文件排序后，重复以上过程，直到最终得到一个大文件，保存到文件output/file.out中，同时生成相应的索引文件output/file.out.index。

Reduce task

  • Shuffle阶段：也称为Copy阶段。Reduce Task从各个Map Task上远程拷贝一片数据，并针对某一片数据，如果其大小超过一定阈值，则写到磁盘上，否则直接放到内存中。

    • 准备运行完成的Map Task列表
      TaskTracker周期性从JobTracker上获取已经运行完成的Map Task列表，并保存到内存中。Reduce Task会周期性通过RPC从TaskTracker上获取已运行完成的Map Task列表，并将成功运行完成的Map Task放到列表mapLocations（TaskTracker Host与已完成任务列表的映射关系）中。避免出现数据访问热点（大量进程集中读取某个TaskTracker上的数据），Reduce Task将所有TaskTracker Host进行混洗操作（随机打乱顺序），然后保存到 scheduledCopies 列表中，从该列表中获取待拷贝的Map Task输出数据位置。

    • 远程拷贝数据
      Reduce Task同时启动5个（默认值）数据拷贝线程。该线程从scheduledCopies列表中获取Map Task数据输出描述对象，并利用HTTP Get从对应的TaskTracker远程拷贝数据。针对某一片数据，如果其大小超过一定阈值，则写到磁盘上，否则直接放到内存中。

      当出现拷贝错误时，Reduce Task提供了以下几个容错机制：

      • 采用对数回归模型推迟一段时间后重新拷贝对应MapTask的输出数据
      • 当拷贝数据出错次数到达一个值，进行一些检查，以决定是否要杀死该reduce task；
      • 当拷贝数据出错次数继续增加到达一个值，杀死该reduce task，等待调度器重新调度执行
  • Merge阶段：在远程拷贝数据的同时，Reduce Task启动了两个后台线程对内存和磁盘上的文件进行合并，以防止内存使用过多或磁盘上文件过多。
    • LocalFSMerger线程会从磁盘上取出最小的10个（默认值）文件进行合并，并将合并后的文件再次写到磁盘上。
    • InMemFSMergeThread线程会将内存中所有数据合并后写到磁盘上。
  • Sort阶段：用户编写的reduce()函数输入数据是按key进行聚集的一组数据。为了将key相同的数据聚在一起，Hadoop采用了基于排序的策略。由于各个Map Task已经实现对自己的处理结果进行了局部排序，因此，Reduce Task只需对所有数据进行一次归并排序即可。为了提高效率，Hadoop将Sort阶段和Reduce阶段并行化。
  • Reduce阶段：在该阶段中，Reduce Task将每组数据依次交给用户编写的reduce()函数处理。
  • Write阶段：reduce()函数将计算结果写到HDFS上。

Reporter

Task 使用 Reporter 组件周期性向TaskTracker汇报最新进度和计数器值。在Map/Reduce Task中，TaskReporter类实现了Reporter接口，并且以线程形式启动。TaskReporter汇报的信息中包含两部分：任务执行进度和任务计数器值。

当Counter（任务计数器，Hadoop提供的用于实现跟踪任务运行进度的全局计数功能）值或者进度发生变化时，TaskReporter将最新Counter值和进度汇报给TaskTracker；TaskTracker通过心跳将任务的最新Counter值和进度汇报给 JobTracker。

python 内存管理

2019-07-05

python 具有自动管理内存的能力，不需要开发者显式的 申请/释放内存。当一个对象不再被需要，python 可以自动从内存中清除这个对象。但是了解一些 python 对于内存的管理及 GC 总是没什么坏处的，也许能写出更好的代（吹）码（牛）。

内存分配

python 的内存体系结构图如下：

0 层，使用通用内存分配器进行分配，比如使用 C 的 malloc 函数
+1 层，python 的 原生内存分配器 使用 PyMem_的一系列API（比如：PyMem_Malloc、PyMem_Free） 来分配内存，这是个 Python/C API，本质也是对 C 的 malloc/free 等函数的包装
+2 层，使用对象分配器进行内存的分配，默认使用 pymalloc 分配器
+3 层，对于一些内置类型，如 int/dict/list/string，基于速度/空间的权衡，会有一些特别定制的内存分配器。

pymalloc 内存分配器

pymalloc 分配器是为 生命周期短的小对象（小于或等于 512 字节） 而设计的，目的是减少小对象的内存操作开销。对于大于 512 字节的对象，pymalloc 分配器将使用 PyMem_RawMalloc/PyMem_RawRealloc(+1层) 来进行内存分配。

数据结构

pymalloc 主要由三个数据结构表示，从小到大是：block, pool, and arena.

block

block 是固定大小的内存块，每一个 block 放一个大小固定的 python 对象。block 的大小从 8 字节到 512 字节，以 8 的倍数递增，共有 64 种大小的 block。每来一个对象，会把这个对象塞到比这个对象稍大的block中，即下面这张图：

当我们给一个 43 byte 大小的对象分配内存时，就会分一个 48 byte 的 block 给它。

pool

pool 是相同大小 block 的集合，所有共有 64 种。通常一个 pool 的大小是内存页的大小，比如 4KB。相同 block 大小的 pool 通过双向指针全部链接在一起。

每一个 pool 都有三种状态：

  • used（部分使用） — 这个 pool 有部分 block 被使用
  • full（满） — 这个 pool 的所有 block 都已经被使用的
  • empty（空 ）— 这个 pool 没有 block 被使用
arena

介绍 arena 之前，要说明一点，block 和 pool 都不是直接分配内存的，他们用的都是已经分给 arena 的内存。

arena 是一个 256kB 的内存块，为 64 种 pool 提供内存。两个 arena 之间使用双向指针来链接。当有新的内存申请（比如需要两个16 byte 的 pool）时，arena 会自动分配内存。当一个 arena 里符合规格的 pool 都满了无法再分配内存时，就可以再申请一个 arena 了。

内存释放

当一个 block 中的对象不再需要，其所在内存可以被回收时，pymalloc 会将这个 block 重新链接到其对应的 pool的 freeblock 链表中，下次再有符合大小的对象申请内存时，会从对应尺寸的 pool 的 freeblock 链表摘下一个block 来进行分配。

所以，python 的小对象基本是不会将内存还给操作系统的，只有当一个 arena 里所有的 pool 都处于empty状态，这个arena 才会被释放。

pymalloc 总结

当我们使用一些 生命周期很短的小对象时，直接使用 C 的 malloc 和 free 等函数进行内存分配，是会引入较大的系统开销，而且还可能产生大量的内存碎片。

因此，pymalloc 这种类似于内存池的机制被实现出来，用来管理小对象（小于或等于 512 字节）的内存分配。

其缺点也很显然，对于长时间运行的 python 进程，pymalloc 可能会保持很多无用内存。

引用和对象

python 中一切皆对象，还有一个重要概念是动态类型，即变量不需要申明，赋值时，变量可以被赋为任意值。其核心就是引用和对象分离。

不可变对象的引用

字符串、数字、元组是不可变对象。当我们定义了变量指向这些不可变对象时，对变量进行操作后，变量引用的值发生了变化，实际是我们变量指向了新的不可变对象：

可变对象的引用

列表、字典这种容器类对象是可变对象，但他们包含的并不是对象本身，而是对象的引用。

引用次数

每一个 python 对象都是一个 PyObject，包含引用计数器和类型标志符，使用 sys.getrefcount() 可以查看某个对象的被引用次数。

引用次数增加
  • 赋值运算符
  • 函数参数传递
  • 被添加进一个列表（或者其他容器）
引用次数减少
  • 对象别名被 del
  • 对象别名被赋予其他对象
  • 对象离开作用域
    • 全局变量所引用的对象将一直存在于 python 进程中。
    • 块内的变量（比如类、方法），当这个块执行结束后，块里所有被创建的引用都会被删除
  • 对象从容器中被删除或容器被销毁

引用计数法（Reference counting）

思想很简单：一旦某个对象的引用次数变为 0，自动调用该对象的释放函数。该对象释放后，该对象引用的其他对象的引用数会减少。

优点很明显：实时性。对象引用变为 0 后，内存会被立即释放掉。

缺点也很明显：

  • 如果要删除的是一个大字典，需要给大字典里每一个对象的引用数都要减一，可能会花费较长时间。
  • 无法回收循环引用的对象。循环引用的对象的应用次数永远不为 0。

分代回收法（Generational garbage collector）

为了解决引用计数无法回收循环引用对象的问题，python 提供了分代垃圾回收器，也就是 gc 模块。

首先看 python 是怎么进行分代的，基于「越老的对象越不可能是垃圾」的假设，python 将对象分为三代：

  • 第 0 代：所有新生对象都是第 0 代
  • 第 1 代：经过一次垃圾回收后，还活下来的第 0 代对象晋升为第 1 代

  • 第 2 代：经过 一次垃圾回收后，还活下来的第 1 代对象晋升为第 2 代

    垃圾回收时 python 不能进行其他任务，频繁的垃圾回收会大大降低处理效率，因此分代回收要符合一定条件才会进行，条件如下：

  • 收集器保留了自上一次垃圾收集后，对象的分配数和销毁数，当 分配数 - 销毁数 > 700 时，将开启垃圾收集
  • 最开始只有第 0 代被检查并收集
  • 当上一次第 1 代检查后，第 0 代的检查的次数大于阈值 threshold1（10）后，开始检查第 1 代并收集（这时候第 0 代也会被收集）
  • 当上一次第 2 代检查后，当第 1 代检查的次数大于阈值 threshold2（10）后，开始检查第 2 代并收集（这时候第 1 代和第 0 代也会被收集）

CDN 缓存

CDN，中文名称是内容分发网络，可以用来分发直播、网页静态文件、小文件等等，其目的是使用户可就近取得所需内容。缓存场景如下：

  1. A 用户请求服务器的一张图片，请求到达距离最近 CDN 节点（边缘节点），边缘节点检查自己缓存里是否有这张图片，没有则向源站请求这张图片返回给 A 用户并缓存起来。
  2. B 用户请求同一张图片，也落到这个 CDN 节点上，边缘节点检查到有这张图片的缓存，检查后发现没有过期，CDN 节点就直接将该图片返回给 B 用户。
  3. 若过期，则该节点该需要向源站发出回源请求，来拉取最新的图片。将新图片返回给 B 用户并缓存起来。

CDN 节点众多，一方面可以缓解访问压力，另一方面更新缓存数据比较缓慢，一般至少是分钟级别

Read/Write Through

在这种模式下，应用程序只需要关注缓存就可以了，所有读数据和更新数据由缓存（相当于数据库的代理）来执行。

Read Through：应用程序只查询缓存。缓存服务发现取不到数据，缓存服务自己去数据库中加载数据并返回给应用程序。

Write Through ：只写数据到缓存中。缓存服务负责将数据同步写入数据库。缓存服务在实现上应将 cache 和 db 更新放在一个事务中，用来保证数据的一致性。

Write Behind（Write Back）

和 Write Through 不同的是，缓存服务将数据写入到数据库中的过程是异步的。当 缓存中的被修改的数据要被替换时才一次写入数据库中。这种实现方式速度很快，但数据的一致性比较差，还可能会有数据的丢失情况。适合对写入速度有极致要求，而对数据准确性没那么高要求的场景。

Ehcache 实现了上面三种设计模式，可以参考

三种设计模式从上到下复杂度依次上升。还是要具体问题具体对待。

缓存经典问题

缓存穿透

当请求不存在数据时（比如在一个图书管理系统中查询一本不存在的书），缓存未命中，到数据库中去查询，然后返回不存在。一个请求没什么，当高并发的来请求不存在的数据时，所有的流量就都到了数据库，然后数据库就gg 了。

解决方法：

  • 缓存数据库返回的空对象。这样下一个请求到来时，缓存就能直接返回空。占用一定内存。

  • 使用布隆过滤器。布隆过滤器是由一个很长的 bit 数组和一系列 hash 函数组成。在数据库查到结果后，将结果由几个hash函数映射到数组上，例如 f1(aaa)=1,f2(aaa)=4，将bit数组的第二位和第五位置为1，代表 aaa 存在。将所有可能存在的数据映射到布隆过滤器里，当用户访问了一个不存在的值[ f1(noexist)=1,f2(noexist)=6]，布隆过滤器可以迅速返回不存在，避免再到数据库中查询。布隆过滤器无法保证 100% 的正确率。

    下面这张图是不同长度的布隆过滤器在不同个数的哈希函数的误报率对比图。k 为哈希函数个数，m 为布隆过滤器长度，n 为插入的元素个数，p 为误报率。可以看出，当过滤器长度足够大，使用的哈希函数越多，误报率越低。

缓存雪崩

当缓存失效（过期）后，大量请求被转移到数据库，造成数据库压力过重，然后 gg。在高并发的场景下比较容易出现该问题。

解决方法：

  • 加锁等待。只允许一个线程来读数据库更新缓存，其他线程等待或直接返回默认值/空值。
  • 后台更新。不通过业务线程来更新缓存，而通过后台线程定时更新。

缓存的取舍

缓存+高可用数据库

问题：若缓存更新采用 Cache Aside， 数据库做了读写分离，写数据库已是最新数据，读数据库还未同步，那么在这段时间内，读取缓存可能读取的是从库中的老数据。

解决思路一：看系统对于数据不一致的忍受度如何，能接受那就不管它。

解决思路二：缓存更新采用 Write Through，同时更新缓存和写数据库里的值，只要在缓存失效前，读数据库能同步完成即可；需要确认当前的缓存服务是否支持 Write Through 模式，若不支持，还需要改造…

解决思路三：从数据库订阅主数据库数据变动发出的事件，修改从库数据的同时，主动 set 或者 delete 缓存中的数据（看着就感觉很难搞）

缓存的失效时间

  • 过长 –> 很久未被访问的数据还在内存中，浪费内存
  • 过短 –> 应用不断访问数据库拿数据，增大数据库压力
  • 分布式缓存数据的过期时间尽量错开来（加个随机数），避免缓存中很多数据同时失效然后一起去查询数据库，增大数据库压力（缓存雪崩）

进程内缓存+进程外缓存

应用进程内有缓存，比如缓存一下登录的 session，这样会导致应用很难达到「无限扩展」。所以我们会考虑把进程内的缓存丢到进程外去缓存，也就是分布式缓存。

这样在某些情况下也会有问题，考虑极端情况，某条热门微博按照一定的哈希算法缓存在某一台缓存服务器上，然后大家都来看，然后这台缓存服务器被很多应用服务器发过来请求搞挂了，然后流量涌向数据库，然后 gg 了。

在这种情况下，为了缓解那一台缓存服务器的压力，要么将这种热点数据在应用进程内缓存，要么将那台缓存服务器的缓存复制多份，让应用服务器随机选一台访问（听起来还要改造）。

缓存相关测试

对于一个引入了缓存的系统，我们要从哪些角度来进行测试？

  • 验证缓存命中率是一个比较高的值
  • 验证缓存的过期策略，验证缓存过期后是否能取到数据库中新的值
  • 对于进程内的缓存，验证系统正常运行时，分配的缓存空间大小
  • 对于进程外的缓存，验证 添加/减少 缓存服务器，缓存服务正常
  • 对于缓存更新方法，验证数据库与缓存数据的一致性
  • 对于大量的缓存失效，高并发的请求，验证系统能否正常处理
  • 对于大量的缓存穿透，验证系统能否正常处理
  • 对于大量的热点数据访问，验证系统能否正常处理
  • 对于有缓存的系统，验证缓存预热时产生的数据库压力应小于数据库可承受的压力

总结

首先我们了解了缓存的必要性，根据「二八原则」将访问次数多的、变化少的数据放进缓存，存在用户的浏览器、CDN、服务器代理、应用服务器、缓存服务器等位置。

对于缓存什么时候生成/更新，我们了解了三种设计模式：Cache Aside、Read/Write Through、Write Behind。

任何技术在使用过程中，随着场景的复杂度提高，都会出现一些问题，缓存也不例外，缓存中两个经典问题：「缓存穿透」和「缓存雪崩」。

缓存的设计要针对每个系统一一定制，没有一个大而全的方案，但是只要是好用的方案就是好方案。针对这些缓存的设计，提出了一些验证方案。

关于缓存，还有一些本文没有涉及，比如缓存监控、各种缓存技术的实现等等。道阻且长，好好学习。

robotium native的顺序遍历

2019-03-18

功能

使用 robotium 按照一定的顺序自动点击完 Android app 所有的白名单 native 控件，可用于稳定性测试

概述

主要步骤为：

  1. 每新到一个activity，找到当前 activity 所有的白名单控件，保存在一个 map 中(PageStore Map<String, PageContent>)
  2. 点击 activity 的未点击的控件，点击
  3. 重复1、2步，直到所有的页面都没有未点击的控件

由于listview 和 recyclerview 中大部分元素点击是一样效果，此处进行了省略，按 Item 进行点击，即一个列表有三项，则点击三次

主要组件

PageContent

存储 Activity 的相关控件信息

  • String activityName - 页面的唯一标志

  • HashSet selectSet - 未点击集合（过滤listview和recycleview中的view）

  • HashSet noSelectSet - 已点击集合/黑名单集合

  • HashSet optionSet - 当一些控件在页面上找不到时可以点击的元素（点了之后可能找到控件）

  • Map<String,Integer> listMap - 页面上 listview 的集合

    记录格式为：viewpageritem#pagepos#item : postion

    viewpageritem：一个 activity 里可能有多个viewpager，viewpageritem 代表getCurrentView获得的viewpager 的 index。viewpageritem=-1代表无viewpager

    pagepos：listview 在 viewpager 的第几页

    item：getCurrentView获得的 listview 的 index

    postion：listview已经点到了第几个。position=-1列表已点击完毕

    | 1 | n（对应关系） |
    | activity | viewpager |
    | viewpager | pager |
    | pager | listview |
    | listview | listItem |

  • Map<String,Integer> recylerViewMap - 页面上 recylerView 的集合

    记录格式为：viewpageritem#pagepos#item : postion

  • HashSet failSet - 因找不到控件而忽略点击的控件集合

  • boolean retry，int failnum - 失败重试机制，failnum 超过一定次数刷新页面（认为当前页面结构已有变动）

PageStore

生成 pageContent 的类，定义一些设置

  • Solo solo、Instrumentation instrumentation - 工具类
  • Class[] whiteClass - 白名单类，只获取/点击白名单类的控件，比如{TextView.class, ImageView.class}
  • String[] backBtns - 返回键的 resourceId，这些控件不会被点击（当一个页面只有一个入口可以进入，点击该页面返回后，该页面就不会再被进入到，影响页面遍历），可以当做黑名单控件来用
  • int[] backBtnIds - 上面数组对应的resourceId
  • String loadingBtns - loading 的 resourceId，用于确保页面加载完成
  • Map<String, PageContent> activityMap - 用于存储遍历的每一个页面
  • IniUtil iniUtil - 用来读取 ini 配置文件的工具类

功能流程

找控件 - 构造PageContent

这一步骤是为了获取当前 activity 中所有的白名单控件，通常使用 solo.getCurrentViews() 来获取当前页面控件。

在复杂的页面布局下，使用该方法无法获取该 Activity 所有的控件，因此，此处主要按照 ViewPager –> ListView/RecyclerView –> ScrollView 的处理顺序来获取当前页面所有的白名单控件。

此处简单控件是指非 listView/recyclerView 的其他控件。

找控件 - 处理scrollView

solo 的 getCurrentViews 不能获取未展示在屏幕范围内的 view，一些过长的列表元素无法全部获取，对可滑动视图进行处理，将其滑动到底部，再使用 getCurrentViews 以获得所有控件

点击控件 - 判断控件是否在屏幕上展示

此步骤主要是为了点击我们想要的控件，而不是点击覆盖在该控件上的其他控件。判断控件的实际x坐标是处于[0，屏幕x轴像素] 之间，y坐标处于[0，屏幕y轴像素]之间。

点击控件 - 处理listview/recyclerview点击

为了保证点击的列表项是展示在屏幕上的，此处根据 viewpageritem#pagepos#item : postion，将viewPager设定为正确的page页（pagepos），再根据postion对列表项进行点击

点击失败刷新机制

当多次找不到某个控件时，确认当前 activityname 等于我们寻找的控件所属 pageContent 的 activityname，即，我们是在正确的页面查找控件。

否则，将失败次数重置，直接返回，不做处理

activityname 相等时，则直接重构该 PageContent 的可点击控件
